# 多 P 视频支持设计草案

> 状态：仅设计，尚未实现  
> 开发分支：`feature/multi-part-video`  
> 基线版本：`0.10.0`

## 1. 目标

在不改变现有“一个视频任务生成一份完整 Markdown 文档”语义的前提下，支持普通 Bilibili BV 多 P 视频的下载、字幕、ASR、关键帧、总结、缓存、播放和失败回滚。

本设计不支持番剧、电影、纪录片、综艺等 PGC 页面，也不支持课程、活动聚合、音频和直播页面。这些特殊类型继续使用当前稳定版的“暂不支持”逻辑。

## 2. 核心原则

1. 一个 BV 是一个父任务，只生成一份最终 Markdown 文档。
2. 每个分 P 是父任务内部的处理单元，不单独派发给不同 Agent。
3. 一个父任务每次领取仍生成一个全新的 `workId`，所有分 P 共用该次工作尝试。
4. 下载、字幕、ASR、关键帧按分 P 独立执行和记录，AI 在全部材料齐备后统一总结。
5. 最终产物必须覆盖全部分 P。任一必要分 P 缺失时，不接受不完整文档。
6. 中止或失败继续遵守现有规则：清理本次工作缓存、撤销领取、从头重新开始。
7. 已存在于视频缓存库的分 P 文件属于持久缓存，中止工作流时不得误删。
8. 单 P 视频继续走现有路径，避免为了多 P 支持重写稳定流程。

## 3. 数据模型

### 3.1 父视频任务

现有任务继续代表整个 BV，增加以下信息：

- `videoType`: `single-part` 或 `multi-part`
- `pageCount`: 分 P 总数
- `partsSnapshotAt`: 分 P 元数据快照时间
- `partsFingerprint`: 根据 `cid/page/part/duration` 计算的快照摘要
- `multiPartStatus`: `pending/running/complete/failed`

### 3.2 分 P 记录

新增可持久化的 `videoParts` 记录，推荐主键为 `${taskId}:p${page}`：

- `taskId`、`bvid`、`cid`
- `page`: Bilibili 的分 P 序号，从 1 开始
- `title`: 分 P 标题
- `duration`、`dimension`
- `sourceUrl`: 带 `?p=` 的原视频地址
- `downloadStatus`、`subtitleStatus`、`asrStatus`、`frameStatus`
- `mediaPath`、`subtitlePath`、`asrPath`、`framesDir`
- `metadataSnapshot`

分 P 状态用于进度和诊断，不作为可独立派发的 Agent 任务。

### 3.3 元数据变化

每次同步或开始任务前重新读取 `pages`：

- 新增分 P：更新快照，父任务按最新全集处理。
- 删除分 P：保留已完成旧产物；未完成任务按最新快照处理。
- 分 P 改名或排序变化：以 `cid` 识别内容，以 `page` 生成当前展示顺序。
- 工作进行中检测到指纹变化：中止本次工作、清理临时产物并要求重新开始。

## 4. 工作目录

一个父视频仍使用一个视频产物目录：

```text
workspace/用户/收藏夹/视频目录/
  metadata.json
  summary.md
  cover.jpg
  comments.json
  parts/
    P001-分P标题/
      info.json
      merged.mp4
      subtitles/
      asr/
      frames/
      manifest.json
    P002-分P标题/
      ...
```

目录名称中的分 P 序号固定补零。标题必须经过现有安全文件名处理，所有路径继续限制在应用管理的 workspace 内。

## 5. 工具执行流程

1. 读取 BV 总元数据与完整 `pages` 列表。
2. 固化本次任务的分 P 快照与摘要。
3. 为每个分 P 使用对应 `cid` 下载音视频并合轨。
4. 为每个分 P 提取 Bilibili 字幕。
5. 不论是否有自带字幕，都为每个分 P 执行一次 faster-whisper ASR，并保留逐句时间戳。
6. 为每个分 P 提取关键帧，保留分 P 内时间戳。
7. 评论、标签、UP 主、发布日期等 BV 级信息只拉取一次。
8. 全部分 P 材料完成后，生成统一的 Agent 输入清单。
9. Agent 完成总览、思维导图、目录和逐 P 正文。
10. 验证文档覆盖所有分 P 后，原子提交最终产物。

下载和 ASR 继续由资源调度器排队。分 P 可以并发下载，但 GPU ASR 必须服从现有显存资源池，不因一个多 P 视频绕过队列限制。

## 6. Markdown 输出结构

```markdown
# 视频标题

## 小结
## 思维导图
## 目录
## 视频信息与分 P 清单
## P1：分 P 标题
### 本 P 小结
### 正文
## P2：分 P 标题
### 本 P 小结
### 正文
## 跨分 P 综合分析
## 评论分析
## 处理记录
```

所有时间轴链接必须同时包含分 P 和秒数：

```text
https://www.bilibili.com/video/BVxxxxxxxxxx?p=2&t=125
```

图片说明必须标出所属分 P 和该分 P 内时间。文档验证器需要检查分 P 清单、正文覆盖率、链接中的 `p` 参数和所有引用图片是否存在。

## 7. Agent 上下文策略

普通多 P 视频优先在同一个 Agent 任务上下文中完成，以维持跨分 P 的整体理解。

当材料接近模型上下文上限时：

1. 各分 P先生成结构化中间摘要，保留原始材料路径和逐句时间戳索引。
2. 使用同一供应商和模型的新压缩会话，对超长分 P 材料分章节压缩。
3. 原 Agent 接收压缩结果、关键原文引用和分 P 清单后继续生成最终文档。
4. 不把上一条视频任务的对话历史带入新的父任务。

压缩是极端兜底，不应成为普通多 P 视频的默认步骤。

## 8. 缓存与视频库

- 视频缓存收藏夹中的一条 BV 记录可以包含多个分 P 媒体文件。
- 视频库卡片显示“共 N P”，播放器提供分 P 选择列表。
- 选择分 P 后加载对应文件，并保持横屏、竖屏和不同分辨率的正确比例。
- 删除多 P 缓存记录时，必须二次确认并删除该 BV 下全部持久缓存。
- 工作流使用已缓存视频时，校验缓存的分 P 快照；缺少任一分 P 时补齐或中止，不静默降级。

## 9. 失败与回滚

以下情况中止整个父任务，不提交部分文档：

- 任一分 P 下载失败且重试后仍不可用。
- 分 P 列表在处理中发生变化。
- 任一分 P 的必要元数据或媒体无法验证。
- Agent 输出没有覆盖完整分 P 清单。
- 用户停止工作流、应用退出或应用崩溃恢复。

中止后：

- 撤销本次 `workId` 和任务领取状态。
- 清理本次尝试产生的所有分 P 临时文件。
- 保留任务开始前已经存在的持久视频缓存和已完成旧版本产物。
- 任务回到待派发状态，下次生成新 `workId` 并从头处理。
- 日志明确指出失败的分 P、阶段、错误原因和是否清理成功。

## 10. 界面调整

- 任务总览显示“单 P”或“共 N P”。
- Agent 工作流总进度同时展示父任务进度和当前分 P进度。
- 下载视频页面展示每个 BV 的分 P 下载总进度。
- 视频库播放器增加主题化分 P 列表，不使用浏览器默认控件。
- 文档库显示分 P 数量，并允许从目录直接跳转对应 P 的时间轴。
- 多 P 支持正式发布后，原先被标记为 `multi-part` 的未完成任务可重新检测并启用；特殊视频仍保持禁用。

## 11. 兼容与迁移

1. 数据库迁移必须可重复执行，并可在失败时回滚。
2. 现有单 P 任务、文档和视频缓存不转换目录。
3. 现有已完成产物不因新版本重新处理。
4. 现有 `unsupportedKind=multi-part` 的待处理任务，在重新读取元数据成功后解除禁用。
5. `missing-bvid` 和 `special-video` 任务不自动启用。
6. 降级回稳定版时，多 P 任务重新显示为暂不支持，不破坏已完成文档。

## 12. 测试矩阵

合并前至少覆盖：

- 普通单 P 全流程回归。
- 2 P、10 P 和较多分 P 视频。
- 中文、英文、日语及无自带字幕视频。
- 不同分 P 横竖屏比例混合。
- 分 P 同名、特殊字符和超长标题。
- 某个中间分 P 下载失败、无音轨或视频失效。
- ASR 排队、低显存和多个 Agent 并发处理不同 BV。
- 用户手动停止、应用退出、崩溃恢复和缓存清理。
- 已缓存多 P 视频直接派发。
- 分 P 列表在同步前后新增、删除、改名和换序。
- `?p=<page>&t=<seconds>` 链接准确跳转。
- 文档完整性验证、图片路径验证和原子提交。
- 番剧、电影、纪录片、综艺、课程、直播等特殊类型仍被拒绝。

## 13. 合并门槛

只有同时满足以下条件才允许合并到 `main`：

1. 单 P 现有发布门禁全部通过。
2. 多 P 专项单元测试和集成测试全部通过。
3. 至少使用一个真实多 P 视频完成端到端测试。
4. 中止与崩溃测试证明不会留下可被误认成完整产物的文档。
5. 视频库、文档库和 Agent 工作流 UI 完成桌面窗口回归检查。
6. README、DESIGN、部署文档和 Agent 模板同步更新。
7. 经人工确认后再从 `feature/multi-part-video` 合并，不直接在 `main` 开发。

## 14. 附加待办：知识库导出与导入

### 14.1 目标与边界

- [ ] 支持以“一个用户 + 该用户下多选收藏夹”为单位导出完整知识库迁移包。
- [ ] 迁移包包含知识库索引、用户与收藏夹元数据、Markdown 原文、`info.json`、封面、关键帧及文档引用的其它图片资源。
- [ ] 导入完成后，文档立即出现在文档库、RAG 知识库助手、外部只读知识库 API 和导出功能中。
- [ ] 不导出 Cookie、账号密码、AI 供应商密钥、Worker ID、`workId`、运行日志、任务尝试缓存或视频媒体缓存。
- [ ] 不直接复制正在使用的 SQLite 文件；应导出经过版本化的可移植索引，再由导入侧写入自己的数据库。

### 14.2 迁移包格式

建议使用独立扩展名的 ZIP 容器，例如 `*.star-owner-kb.zip`：

```text
manifest.json
index.json
checksums.sha256
users/<source-user-key>/
  collections/<source-collection-key>/
    collection.json
    documents/<document-key>/
      summary.md
      info.json
      cover.*
      frames/
      other-assets/
```

`manifest.json` 至少记录：

- `schemaVersion`、应用版本、`exportedAt` 和导出侧实例标识。
- 用户类型：`bilibili` 或 `builtin`。
- Bilibili 用户 UID、显示名，以及收藏夹稳定 ID、显示名和原始 `storageName`。
- 每个文档的稳定键、BV、发布日期、收藏日期、收藏状态、相对路径和资源清单。
- 每个文件的 SHA-256、字节数和媒体类型。

包内禁止绝对路径、`..`、符号链接和未登记文件。导入前必须在暂存目录校验版本、校验和、文件数量、总大小、扩展名和路径边界。

### 14.3 用户、收藏夹与文档冲突规则

冲突判断顺序固定如下，不能仅依赖可变的显示名：

1. **应用内置用户**：导出侧来源为内置用户时，导入到导入侧现有的内置用户，不创建同级的第二个内置用户。
2. **相同 Bilibili 用户**：导入侧存在相同 Bilibili UID 时，复用该用户记录；显示名可以按导入包更新，但稳定用户 ID 不变。
3. **新的 Bilibili 用户**：UID 不同或导入侧不存在该 UID 时，新建一个与其它 Bilibili 用户同级的用户记录，显示导出侧用户名。即使显示名相同，也必须按 UID 保持为不同用户。
4. **相同远端收藏夹**：仅当 Bilibili UID 与收藏夹稳定 ID 都相同时，才视为同一收藏夹并合并到现有记录。
5. **新收藏夹或内置收藏夹**：在目标用户下新建收藏夹；若名称已存在，名称改为 `<原收藏夹名>（导出于<exportedAt>）`。重复导入仍冲突时再追加递增序号，不能覆盖另一个不同 ID 的同名收藏夹。
6. **相同视频文档**：在已经匹配为同一收藏夹的前提下，以 BV 作为主要稳定键；同一 BV 已有完成文档时，由导入包中的 Markdown、元数据、封面、关键帧和其它关联资源原子覆盖导入侧旧版本，即“导出侧文档优先”。
7. **非重复文档**：作为新的完成知识文档加入目标收藏夹，不影响其它文档和任务。

新导入且无法绑定到当前登录 Bilibili 账号的用户或收藏夹属于“导入知识库来源”，可以正常浏览、RAG、再次导出和通过只读 API 调用，但不自动创建可同步、可派发的视频任务。若导入目标是导入侧已有的同 UID、同收藏夹 ID 记录，覆盖后的文档继续保持该收藏夹原有任务语义和完成状态。

### 14.4 原子导入与恢复

- [ ] 导入分为“解压到暂存区 -> 完整校验 -> 生成冲突计划 -> 用户确认 -> 文件提交 -> SQLite 事务提交”六个阶段。
- [ ] UI 在提交前展示新增用户、复用用户、新增收藏夹、改名收藏夹、新增文档和将被覆盖文档的数量与明细。
- [ ] 覆盖现有文档时，先在同一 Workspace 内生成可回滚备份；数据库提交成功后才清理旧文件与备份。
- [ ] 文件复制、校验、数据库写入或应用崩溃任一环节失败时，恢复导入前状态，并在日志中记录回滚结果。
- [ ] 导入进度需要区分校验、复制、索引写入和最终提交，不能让大型知识库表现为界面卡死。
- [ ] 成功后增量刷新文档库、RAG 目录和知识 API 缓存，不要求重启应用。

## 15. 附加待办：ASR 三模式调度

### 15.1 模式定义

将现有 `cpuAsrEnabled` 布尔配置迁移为明确的模式配置：

- `gpu`：仅启动 NVIDIA CUDA `float16` 常驻 ASR 服务。
- `cpu`：仅启动 CPU `int8` 常驻 ASR 服务，不占用 GPU 显存。
- `hybrid`：同时启动 GPU 与 CPU 常驻服务，两个资源通道并行处理不同 ASR 子任务。

数据库同时保存：

- `requestedMode`：用户选择的模式。
- `effectiveMode`：硬件检测、运行时健康和自动退避后实际运行的模式。
- `fallbackReason`：发生自动退避时的稳定错误码和可读原因。

旧配置迁移规则：`cpuAsrEnabled=false` 映射为 `gpu`，`cpuAsrEnabled=true` 映射为 `hybrid`。迁移必须幂等。

### 15.2 调度语义

- [ ] `gpu` 模式只向 GPU 通道派发；`cpu` 模式只向 CPU 通道派发。
- [ ] `hybrid` 模式允许不同视频或不同分 P 的 ASR 子任务分别占用 GPU 与 CPU 通道，同一音频任务只执行一次，不能在两个设备上重复推理。
- [ ] 混合模式优先使用空闲 GPU；GPU 忙碌、显存门禁不满足或正在退避时，后续任务可进入 CPU 通道，而不是全部阻塞在 GPU 队列后方。
- [ ] GPU 和 CPU 服务使用同一个用户所选模型版本；切换 `small` / `medium` 时两个服务必须一致更新。
- [ ] 模式或模型切换采用排空策略：正在推理的任务先完成，新任务暂停领取，旧服务退出后再启动目标服务，避免强制切换留下半份字幕。

### 15.3 自动退避与恢复

- [ ] 启动时若未检测到兼容 NVIDIA GPU、CTranslate2 CUDA、足够显存或可用 GPU 模型服务，而 CPU 环境满足条件，则 `effectiveMode` 自动切换为 `cpu` 并继续提供 ASR，不阻断视频工作流。
- [ ] GPU 服务连续启动失败、原生运行库异常或达到现有连续失败门槛时，从 `gpu` 自动退避到 `cpu`；`hybrid` 模式则停用故障 GPU 通道并由 CPU 继续处理队列。
- [ ] GPU 任务在尚未产生可接受产物前发生基础设施故障时，可以从头转交 CPU 重试；不得拼接两个设备各自产生的半份字幕。
- [ ] CPU 环境也不满足模型内存、Windows x64、运行时或模型完整性要求时，停止领取需要 ASR 的新任务，并返回具体诊断，不允许静默跳过 ASR。
- [ ] 自动退避不覆盖 `requestedMode`。用户执行“重新检测”或下次启动时重新检查 GPU；恢复条件满足后再回到请求模式，避免故障期间频繁来回切换。
- [ ] UI、日志和 Agent 工作记录必须同时显示请求模式、实际模式、退避原因、服务 PID、模型、设备和每次任务实际使用的通道。

### 15.4 界面与兼容性

- [ ] 设置页使用三段式模式选择器：`仅 GPU`、`仅 CPU`、`GPU + CPU`。
- [ ] 不可用模式置灰并展示缺失条件；GPU 不可用但 CPU 可用时，界面明确显示“已自动使用 CPU”，而不是把整个 ASR 标记为不可用。
- [ ] 工具状态页分别展示 GPU/CPU 服务状态、队列长度、处理数量、平均耗时、失败次数和最近退避记录。
- [ ] 默认模式保持 `gpu`，但首次检测失败时按上述规则自动使用 CPU；CPU 模式仍允许用户手动选择 `small` 降低内存和处理时间。

### 15.5 测试矩阵

- [ ] 三种模式分别使用真实短音频生成 SRT、时间轴文本和结构化 JSON。
- [ ] `hybrid` 同时提交至少两个 ASR 任务，证明 GPU 与 CPU 进程并行工作且无重复任务。
- [ ] 无 NVIDIA GPU、CUDA 不可见、显存不足、GPU 服务启动失败时自动退避 CPU。
- [ ] CPU 也不可用时阻止工作流并返回稳定错误码和可读原因。
- [ ] 模式切换、模型切换、应用退出、崩溃恢复和任务中止不会遗留服务进程或半成品。
- [ ] 多 Agent、多 P 视频和缓存视频并发时仍遵守资源池、任务回滚和缓存保护规则。
- [ ] 旧 `cpuAsrEnabled` 配置升级后模式正确，重复迁移不会改变用户选择。

## 16. 分支实施顺序补充

以上两项与多 P 支持在同一开发分支记录，但应拆分为独立提交和独立测试门禁：

1. 正式开发前先同步最新 `main`，解决设计分支与当前稳定版的差异，并通过未修改功能的发布基线测试。
2. 先完成知识库迁移包 schema、冲突计划和安全校验，再接入 UI 与原子导入。
3. 再完成 ASR 配置迁移、三模式资源池和自动退避，并回归现有 GPU 与 CPU 单通道。
4. 最后将多 P 子任务接入新的 ASR 三模式调度，执行完整并发与回滚测试。
5. 三项功能全部通过各自测试矩阵和现有 `verify:release` 后，才进入人工合并评审。
