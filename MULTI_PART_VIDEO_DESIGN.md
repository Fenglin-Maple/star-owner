# 多 P 视频支持设计草案

> 状态：仅设计，尚未实现  
> 开发分支：`feature/multi-part-video`  
> 基线版本：`0.10.0`

## 1. 目标

在不改变现有“一个视频任务生成一份完整 Markdown 文档”语义的前提下，支持普通 Bilibili BV 多 P 视频的下载、字幕、ASR、关键帧、总结、缓存、播放和失败回滚。

本设计不支持番剧、电影、纪录片、综艺等 PGC 页面，也不支持课程、活动聚合、音频和直播页面。这些特殊类型继续使用当前稳定版的“暂不支持”逻辑。

## 2. 核心原则

1. 一个 BV 是一个父任务，只生成一份最终 Markdown 文档。
2. 每个分 P 是父任务内部的处理单元，不单独派发给不同 Agent。
3. 一个父任务每次领取仍生成一个全新的 `workId`，所有分 P 共用该次工作尝试。
4. 下载、字幕、ASR、关键帧按分 P 独立执行和记录，AI 在全部材料齐备后统一总结。
5. 最终产物必须覆盖全部分 P。任一必要分 P 缺失时，不接受不完整文档。
6. 中止或失败继续遵守现有规则：清理本次工作缓存、撤销领取、从头重新开始。
7. 已存在于视频缓存库的分 P 文件属于持久缓存，中止工作流时不得误删。
8. 单 P 视频继续走现有路径，避免为了多 P 支持重写稳定流程。

## 3. 数据模型

### 3.1 父视频任务

现有任务继续代表整个 BV，增加以下信息：

- `videoType`: `single-part` 或 `multi-part`
- `pageCount`: 分 P 总数
- `partsSnapshotAt`: 分 P 元数据快照时间
- `partsFingerprint`: 根据 `cid/page/part/duration` 计算的快照摘要
- `multiPartStatus`: `pending/running/complete/failed`

### 3.2 分 P 记录

新增可持久化的 `videoParts` 记录，推荐主键为 `${taskId}:p${page}`：

- `taskId`、`bvid`、`cid`
- `page`: Bilibili 的分 P 序号，从 1 开始
- `title`: 分 P 标题
- `duration`、`dimension`
- `sourceUrl`: 带 `?p=` 的原视频地址
- `downloadStatus`、`subtitleStatus`、`asrStatus`、`frameStatus`
- `mediaPath`、`subtitlePath`、`asrPath`、`framesDir`
- `metadataSnapshot`

分 P 状态用于进度和诊断，不作为可独立派发的 Agent 任务。

### 3.3 元数据变化

每次同步或开始任务前重新读取 `pages`：

- 新增分 P：更新快照，父任务按最新全集处理。
- 删除分 P：保留已完成旧产物；未完成任务按最新快照处理。
- 分 P 改名或排序变化：以 `cid` 识别内容，以 `page` 生成当前展示顺序。
- 工作进行中检测到指纹变化：中止本次工作、清理临时产物并要求重新开始。

## 4. 工作目录

一个父视频仍使用一个视频产物目录：

```text
workspace/用户/收藏夹/视频目录/
  metadata.json
  summary.md
  cover.jpg
  comments.json
  parts/
    P001-分P标题/
      info.json
      merged.mp4
      subtitles/
      asr/
      frames/
      manifest.json
    P002-分P标题/
      ...
```

目录名称中的分 P 序号固定补零。标题必须经过现有安全文件名处理，所有路径继续限制在应用管理的 workspace 内。

## 5. 工具执行流程

1. 读取 BV 总元数据与完整 `pages` 列表。
2. 固化本次任务的分 P 快照与摘要。
3. 为每个分 P 使用对应 `cid` 下载音视频并合轨。
4. 为每个分 P 提取 Bilibili 字幕。
5. 不论是否有自带字幕，都为每个分 P 执行一次 faster-whisper ASR，并保留逐句时间戳。
6. 为每个分 P 提取关键帧，保留分 P 内时间戳。
7. 评论、标签、UP 主、发布日期等 BV 级信息只拉取一次。
8. 全部分 P 材料完成后，生成统一的 Agent 输入清单。
9. Agent 完成总览、思维导图、目录和逐 P 正文。
10. 验证文档覆盖所有分 P 后，原子提交最终产物。

下载和 ASR 继续由资源调度器排队。分 P 可以并发下载，但 GPU ASR 必须服从现有显存资源池，不因一个多 P 视频绕过队列限制。

## 6. Markdown 输出结构

```markdown
# 视频标题

## 小结
## 思维导图
## 目录
## 视频信息与分 P 清单
## P1：分 P 标题
### 本 P 小结
### 正文
## P2：分 P 标题
### 本 P 小结
### 正文
## 跨分 P 综合分析
## 评论分析
## 处理记录
```

所有时间轴链接必须同时包含分 P 和秒数：

```text
https://www.bilibili.com/video/BVxxxxxxxxxx?p=2&t=125
```

图片说明必须标出所属分 P 和该分 P 内时间。文档验证器需要检查分 P 清单、正文覆盖率、链接中的 `p` 参数和所有引用图片是否存在。

## 7. Agent 上下文策略

普通多 P 视频优先在同一个 Agent 任务上下文中完成，以维持跨分 P 的整体理解。

当材料接近模型上下文上限时：

1. 各分 P先生成结构化中间摘要，保留原始材料路径和逐句时间戳索引。
2. 使用同一供应商和模型的新压缩会话，对超长分 P 材料分章节压缩。
3. 原 Agent 接收压缩结果、关键原文引用和分 P 清单后继续生成最终文档。
4. 不把上一条视频任务的对话历史带入新的父任务。

压缩是极端兜底，不应成为普通多 P 视频的默认步骤。

## 8. 缓存与视频库

- 视频缓存收藏夹中的一条 BV 记录可以包含多个分 P 媒体文件。
- 视频库卡片显示“共 N P”，播放器提供分 P 选择列表。
- 选择分 P 后加载对应文件，并保持横屏、竖屏和不同分辨率的正确比例。
- 删除多 P 缓存记录时，必须二次确认并删除该 BV 下全部持久缓存。
- 工作流使用已缓存视频时，校验缓存的分 P 快照；缺少任一分 P 时补齐或中止，不静默降级。

## 9. 失败与回滚

以下情况中止整个父任务，不提交部分文档：

- 任一分 P 下载失败且重试后仍不可用。
- 分 P 列表在处理中发生变化。
- 任一分 P 的必要元数据或媒体无法验证。
- Agent 输出没有覆盖完整分 P 清单。
- 用户停止工作流、应用退出或应用崩溃恢复。

中止后：

- 撤销本次 `workId` 和任务领取状态。
- 清理本次尝试产生的所有分 P 临时文件。
- 保留任务开始前已经存在的持久视频缓存和已完成旧版本产物。
- 任务回到待派发状态，下次生成新 `workId` 并从头处理。
- 日志明确指出失败的分 P、阶段、错误原因和是否清理成功。

## 10. 界面调整

- 任务总览显示“单 P”或“共 N P”。
- Agent 工作流总进度同时展示父任务进度和当前分 P进度。
- 下载视频页面展示每个 BV 的分 P 下载总进度。
- 视频库播放器增加主题化分 P 列表，不使用浏览器默认控件。
- 文档库显示分 P 数量，并允许从目录直接跳转对应 P 的时间轴。
- 多 P 支持正式发布后，原先被标记为 `multi-part` 的未完成任务可重新检测并启用；特殊视频仍保持禁用。

## 11. 兼容与迁移

1. 数据库迁移必须可重复执行，并可在失败时回滚。
2. 现有单 P 任务、文档和视频缓存不转换目录。
3. 现有已完成产物不因新版本重新处理。
4. 现有 `unsupportedKind=multi-part` 的待处理任务，在重新读取元数据成功后解除禁用。
5. `missing-bvid` 和 `special-video` 任务不自动启用。
6. 降级回稳定版时，多 P 任务重新显示为暂不支持，不破坏已完成文档。

## 12. 测试矩阵

合并前至少覆盖：

- 普通单 P 全流程回归。
- 2 P、10 P 和较多分 P 视频。
- 中文、英文、日语及无自带字幕视频。
- 不同分 P 横竖屏比例混合。
- 分 P 同名、特殊字符和超长标题。
- 某个中间分 P 下载失败、无音轨或视频失效。
- ASR 排队、低显存和多个 Agent 并发处理不同 BV。
- 用户手动停止、应用退出、崩溃恢复和缓存清理。
- 已缓存多 P 视频直接派发。
- 分 P 列表在同步前后新增、删除、改名和换序。
- `?p=<page>&t=<seconds>` 链接准确跳转。
- 文档完整性验证、图片路径验证和原子提交。
- 番剧、电影、纪录片、综艺、课程、直播等特殊类型仍被拒绝。

## 13. 合并门槛

只有同时满足以下条件才允许合并到 `main`：

1. 单 P 现有发布门禁全部通过。
2. 多 P 专项单元测试和集成测试全部通过。
3. 至少使用一个真实多 P 视频完成端到端测试。
4. 中止与崩溃测试证明不会留下可被误认成完整产物的文档。
5. 视频库、文档库和 Agent 工作流 UI 完成桌面窗口回归检查。
6. README、DESIGN、部署文档和 Agent 模板同步更新。
7. 经人工确认后再从 `feature/multi-part-video` 合并，不直接在 `main` 开发。
